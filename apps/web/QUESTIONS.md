# apps/web — 질문·보류·불가피한 결정 기록

> WORKFLOW §3 / conventions §5-3. 명세가 모호하거나 기존 관측면 수정이 불가피할 때 사유를 남긴다.

## M6-A 비모달 인플레이스 전환 (docs/specs/m6a-inplace-sim.md, #55)

### M6-A AW-31 좌표 재계산 — 기존 e2e 수정 사유 (불가피)

- **무엇**: `e2e/m5-d-web.spec.ts`의 `AW-31 호버 시 고스트 표시, 이탈 시 숨김` 테스트에서, 도구
  버튼(`tool-erase`·`tool-cell`) 클릭 뒤의 마우스 이동 좌표를 한 번 캡처한 `pt` 재사용 대신 매
  클릭 후 `pagePoint`로 다시 계산하도록 바꿨다. 관측(고스트 표시/은닉)·`data-testid`·테스트 이름
  (AW-31)은 그대로다.
- **왜 불가피**: AW-34가 규정한 대로 오버레이 모달(`position: fixed; inset: 0`)을 제거하면 편집
  영역이 문서 흐름에 놓인다. 보드(572px)+상단 헤더로 편집 뷰가 뷰포트(1280×720 테스트 기준)보다
  높아, 보드 아래 도구 버튼을 클릭하면 브라우저가 버튼을 뷰포트로 스크롤한다(실측: 클릭 시 147px
  스크롤 → 보드가 위로 이동). 모달일 때는 fixed 오버레이가 사이트 헤더를 덮고 92vh 안에 들어와
  스크롤이 없었기에 `pt` 재사용이 성립했으나, **비모달 전환의 본질(오버레이 제거 → 문서 흐름 →
  스크롤)** 상 이 가정은 유지될 수 없다. 재생 모드도 동일하게 스크러버가 접힘선 아래에 있어
  스크롤되지만, 다른 테스트는 좌표를 매번 새로 계산하므로 영향이 없다 — AW-31만 유일하게 클릭을
  가로질러 좌표를 재사용했다.
- **대안 검토**: 팔레트를 보드 옆/위로 옮기면 스크롤을 피하지만 AW-35("재생 슬라이더 자리 = 보드
  아래")와 어긋난다. 헤더를 줄여 뷰포트에 맞추는 방식은 사이트 헤더 높이·뷰포트에 의존해 취약하다
  (헤더가 몇 px만 바뀌어도 재발). 따라서 구현은 명세에 충실하게 유지(팔레트는 보드 아래)하고,
  테스트의 좌표 재사용만 고쳤다. 이 변경은 어떤 단언도 약화하지 않으며, 오히려 erase 도구 검사가
  실제 셀 위에서 이뤄지도록 바로잡는다.
- **상태**: 소유자 확인 대기. 반려 시 팔레트 배치(보드 아래 vs 보드 옆) 재론.
