# 결정 기록 (Decisions)

이 문서는 **코드를 읽어서는 알 수 없는 것**만 담는다. 되돌리기 어렵거나, 되돌리려면 근거를 다시 따져야 하는 결정들이다.

## 이 문서의 규칙

- **여기 적힌 것은 승인된 결정이다.** 추측·미결·"아마 이래서 그랬을 것"은 들어오지 않는다. 그런 것은 GitHub 이슈로 간다.
- **코드가 강제하는 값은 적지 않는다.** 한도·상수·스키마 타입의 유일 출처는 `packages/types`다. 여기에 숫자를 복사하면 두 곳이 갈라진다.
- **일정·순서는 적지 않는다.** 그것은 GitHub Milestone 소관이다.
- **결정을 뒤집으면 지우지 말고 `~~취소선~~` + 대체 항목 번호를 남긴다.** 뒤집힌 이력 자체가 정보다.
- 항목 번호는 영구적이다. 재사용하지 않는다.

> **날짜 표기 없음**: 아래 결정 대부분은 MVP 개발 중(2026-07 이전)에 내려졌으나 정확한 시점 기록이 없다. 지어내지 않고 비워 둔다. 이후 추가되는 항목부터 날짜를 적는다.

## 저장 · 공유

### D-1. 저장소는 GitHub Gist, 접근은 Cloudflare Workers 프록시 경유

전용 도메인·유료 서버 없이 운영하는 것이 제약이었다. 무료 저장소들을 직접 조사한 뒤 확정했다. 전용 GitHub 부계정의 PAT를 Workers 시크릿으로 보관하고, 클라이언트에는 토큰을 노출하지 않는다. 조회는 Workers를 거치지 않고 Gist에 직접 요청한다.

*결정자: 소유자 (기획서 초안 시점 확정)*

### D-2. 회원 기능 없음

누구나 접속 즉시 리플레이·노트를 보고 만들 수 있어야 한다. 접근성이 코칭·피드백이라는 서비스 목적의 전제다.

사용자 개인 GitHub 계정 OAuth(각자 자기 Gist에 저장)는 채택하지 않는다. 기획 구체화 중 대화가 잠시 그 방향으로 흘렀으나 접근성 문제로 즉시 정정했다.

**따름정리:** 사용자 계정이 없으므로 데이터 마이그레이션은 소유자가 PAT로 직접 수행한다. "사용자에게 다시 올려달라"는 선택지가 없다.

*결정자: 소유자*

### D-3. 리플레이와 meta는 불변. 노트만 편집 가능

**오답노트 모델.** 코칭·피드백이 목적이므로 오답에 해당하는 원본이 반드시 보존되어야 한다. 리플레이를 편집 가능하게 하면 서비스 목적과 어긋난다.

노트의 버전 관리 모델(branch/merge 등)도 검토했으나 채택하지 않았다. 일반 사용자 기준 복잡성만 증가하고, 그것을 원할 극소수를 위한 과잉 설계다.

**이 원칙은 다른 결정을 구속한다:** 노트의 진입점은 불변 대상(리플레이)만 가리킬 수 있다. → D-8

*결정자: 소유자*

### D-4. 노트 편집 권한은 클라이언트 생성 시크릿 + SHA-256 해시 대조

노트 생성 시 클라이언트가 `crypto.getRandomValues` 기반 시크릿을 만들고, 해시만 노트에 저장한다. 원본 시크릿은 생성자 브라우저 쿠키에만 있다. 쿠키를 잃으면 수정 불가 — 이는 의도된 트레이드오프다.

서버 발급 토큰 방식도 검토했으나 채택하지 않았다. 토큰 역시 브라우저에 저장되므로 위협 모델이 동일하고, Workers에 상태 저장소만 추가된다.

**안전성의 근거는 오직 난수 엔트로피다.** 해시가 Gist에 공개되므로 공격자는 오프라인에서 무제한 속도로 역산을 시도할 수 있다(rate limit이 걸리지 않는다). CSPRNG를 벗어나면 이 방식은 즉시 무너진다. **`Math.random` 계열로 대체 금지.**

*결정자: 소유자 (원안) / 서버 토큰 대안 검토 후 유지 재확인*

### D-5. 노트 파일은 기여자당 1개 (`notes-<clientId>.json`)

노트당 1파일은 하나의 Gist에 파일이 지나치게 많아질 수 있다. 전체 1파일은 동시 수정 충돌 범위가 Gist 전체로 커진다. 기여자 단위가 절충점이다.

*결정자: 에이전트 원안 / 소유자 재승인*

### D-6. 테스트 fixture는 익명화하여 공개 저장소에 포함

골든 테스트(E-3/4/5)는 `@haelp/teto` 실행 결과와 대조해 엔진을 강제한다. CI는 매번 새로 클론된 환경에서 돌므로, fixture가 저장소에 없으면 테스트가 skip되고 **"다르면 triangle이 이긴다"는 엔진의 핵심 보증이 CI에서 검증되지 않는다.** 로컬만 green이고 CI는 빈 껍데기가 된다.

세 안(공개 / 익명화 공개 / 비공개+skip) 중 비공개가 탈락한 이유가 이것이다.

fixture는 전부 소유자 본인의 리플레이이며, 닉네임 등 식별 정보가 제거된 상태임을 파일 단위로 확인했다. 저장소가 공개인 것은 D-17의 결과다.

*결정자: 소유자 (근거는 사후 확인)*

## 데이터 모델

### D-7. 페이지는 조작 로그가 아니라 상태 체크포인트

페이지는 **보드 상태 + 홀드 + 넥스트 큐 + 주석**의 스냅샷이다. 사용자가 명시적으로 저장할 때만 생성된다.

초기에 에이전트가 제안한 스키마는 `Op`(`draw` / `place`) 필드를 포함한 델타/조작 로그 모델이었고, 그 결과 **미노 배치 하나·셀 그리기 하나마다 페이지가 생성**됐다. 노트를 만드는 쪽도 불편하고, 코칭·피드백을 받는 쪽도 넘겨야 할 페이지가 과도했다. 기획 의도(페이지 = 의미 있는 국면)와 어긋나 폐기했다.

**함께 포기한 것:** 조작 순서를 저장하지 않으므로 조작 재생 애니메이션은 불가능하다. 구현 난이도와 사용성 모두 정당화되지 않아 포기했다.

*결정자: 소유자 (에이전트 원안 반려)*

### D-8. 노트의 진입점(`origin`)은 항상 리플레이. fork는 참조가 아니라 복사

다른 사람의 노트에서 시작해 자신의 노트를 만드는 기능(fork)은 기획에 포함된다. 그러나 **구현은 원본 노트를 참조하는 방식이 아니라, 진입 스냅샷을 복사하는 방식이다.**

이유: 노트는 가변이다(D-3). `origin`이 다른 노트의 페이지를 가리키면, 원본 작성자가 그 페이지를 수정·삭제할 때 fork의 진입점이 증발한다. 원본이 또 fork일 경우 참조 체인이 생겨 해석에 다중 fetch가 필요해진다. **"불변 대상만 가리킨다"는 D-3의 원칙을 깨뜨린다.**

출처 표시가 필요하면 진입점이 아니라 **정보성 필드**로 기록한다. 원본이 사라져도 "출처 불명"일 뿐 노트는 정상 동작해야 한다.

**따름정리:** `origin`이 항상 `{round, player, frame}`이므로 참조 체인이 없다. 노트 id를 내용 기반으로 파생할 이유가 사라진다 — 같은 프레임에서 두 사람이 각자 노트를 쓰는 것은 중복이 아니라 정상이므로, 내용 기반 파생의 이점(중복 제거)이 성립하지 않는다.

*결정자: 소유자 (기능) / 구조는 2026-07 재검토에서 확정*

### D-9. 더미 셀(`"D"`)과 overlays는 v1에 필드만 예약

더미 셀은 소유자 원안. overlays는 four-tris의 아이디어를 참고했다. 둘 다 저작 UI 없이 스키마 예약만 한다.

*결정자: 소유자*

### D-10. 카운터의 `-1` 규약을 전 모듈이 공유

정규화 계층을 두지 않고 규약 자체를 공유한다. 실제 오프바이원 사고가 있었던 것은 아니며 예방적 판단이다.

*결정자: 에이전트 원안 / 소유자 재승인*

## 엔진

### D-11. 엔진은 둘. triangle(재생 전용)과 자체 엔진

Tetr.io 리플레이 재생에는 triangle이 필수다. 자체 엔진을 별도로 두는 이유:

- Jstris·Tetr.io에 없는 기능으로 확장할 여지 (가이드 도구, 스태킹 봇, 시뮬레이터)
- **두 플랫폼의 단위 시간이 다르다.** Tetr.io는 frame + subframe, Jstris는 ms. 하나의 엔진으로 통일하려면 어느 한쪽의 시간 모델을 강요해야 한다.

*결정자: 소유자 (에이전트와 논의 후 최종 판단)*

### D-12. 어댑터는 단방향. 자체 엔진 상태 → 플랫폼 포맷 변환은 하지 않음

재생만 하면 되므로 역방향이 불필요하다.

기획서 초안에서 리플레이 간 변환(Jstris ↔ Tetr.io)도 고려했으나 두 이유로 포기했다:

1. 단위 시간이 달라 손실 없는 변환이 불가능하다 (D-11)
2. **Jstris는 서버 오류 시 관리자에게 요청해 리플레이 수동 등록이 가능하다.** 변환 기능이 부정행위에 악용될 수 있다.

**2번이 이 결정의 핵심이며, 되돌리려면 이 위험을 다시 따져야 한다.**

*결정자: 소유자 (도메인 지식 기반)*

### D-13. 엔진 API는 원자 조작 기반

사용자 키 입력을 엔진에 직결하지 않고 입력 레이어를 경유시킨다.

근거: **Jstris 리플레이는 원자 조작(`MOVE_LEFT`, `DAS_LEFT` 등) 단위로 저장된다.** 역공학, Jstris 개발자의 커뮤니티 발언, 공개된 Jstris 리플레이 도구 저장소 세 경로로 확인된 사실이다. 향후 Jstris를 지원하려면 조작 단위 재생이 필요하므로 이 구조가 요구된다.

*결정자: 에이전트 원안 / 소유자가 근거 확인 후 재승인*

### D-14. 지원 룰셋의 경계 — 메이저 팬 룰까지, 마이너 커스텀은 미지원

소유자의 테트리스 도메인 판단이다.

미지원 룰의 처리는 비대칭이다:

- **차단**: 특정 조작 시 미노의 최종 위치를 판단할 수 없게 만드는 룰 (예: 커스텀 킥테이블). 재생 자체가 불가능하므로 사유와 함께 거부한다.
- **경고 + 대체**: 지원 룰로 근사 가능한 것 (예: 스핀 판정 모드). 배지로 알리고 재생한다.

경계선은 **"조작 가능성이 바뀌는가"**다.

`spinBonuses: "all"`(all-spin)은 마이너 룰 중 사용자가 가장 많아, 지원 확대 시 최우선 후보다.

*결정자: 소유자 (도메인 지식 기반)*

## 플랫폼 · UI · 배포

### D-15. Tetr.io 우선, Jstris 후순위

Jstris는 리플레이 포맷 역공학이 선행되어야 해 개발 공수가 크다. 플레이어 수와 접근성 모두 Tetr.io가 우선이다.

**이 결정은 "Jstris를 하지 않는다"가 아니다.** D-11·D-12·D-13은 모두 Jstris 지원을 전제로 내려졌다. 순서만 뒤다.

*결정자: 소유자*

### D-16. FE 프레임워크는 Astro

다른 FE 프레임워크를 추가하거나 혼용할 수 있다는 점 때문에 선택했다.

**근거의 한계를 명시한다:** Astro만 소유자가 정했고, Preact·캔버스 렌더링 방식은 에이전트 판단을 그대로 수용했다. 소유자에게 FE 도메인 지식이 없어 대안을 비교하지 않았다. 별도의 FE·UI 개선 마일스톤에서 재론 대상이다.

*결정자: 소유자 (Astro) / 에이전트 (나머지, 미검증)*

### D-17. 배포 대상은 조직 루트 사이트 (`tetorial.github.io`)

프로젝트 페이지가 아니라 조직 루트 사이트다. 초기에 에이전트에게 `tetorial`이 저장소명인 것으로 잘못 전달되어 프로젝트 페이지로 진행됐다가 정정했다.

**따름정리:** 무료 플랜에서 GitHub Pages를 쓰려면 저장소가 공개여야 한다. D-6의 fixture 공개는 이 결정의 결과다.

*결정자: 소유자*

## 프로세스

### D-18. 커리큘럼(가이드 축) 설계는 위임 불가

에이전트보다 소유자의 테트리스 도메인 지식이 높다. 커리큘럼은 소유자가 내용을 작성하고, 에이전트는 표현을 다듬거나 문장을 교정하는 보조 역할로만 사용한다.

*결정자: 소유자*
