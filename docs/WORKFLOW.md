# 작업 방식 (Workflow)

이 프로젝트는 솔로 프로젝트지만 규모상 하나의 세션으로 진행하지 않는다. **총괄 세션이 계획하고, 작업 세션이 구현한다.** 이 문서는 그 규약이다.

MVP(W0~W4)를 이 방식으로 완주했고, 이 문서는 그 과정에서 실제로 뚫린 지점을 반영해 개정된 버전이다. 각 규칙 뒤에 왜 그것이 있는지 적어 뒀다. **근거 없이 규칙을 줄이지 마라.** 아래 대부분은 사고가 나고 나서 추가된 것이다.

---

## 1. 역할

| 역할 | 담당 |
|---|---|
| **소유자** | 무엇을 만들지 결정. 수용 기준 승인. 게이트 최종 판정. 테트리스 도메인 판단 전권. |
| **총괄 세션** | 소유자와 계획 수립. 웨이브 분할. 명세·프롬프트 작성. 게이트 검사. |
| **작업 세션** | 명세 하나를 받아 패키지 하나를 구현. 담당 밖은 건드리지 않음. |

**세션을 분리하는 이유는 컨텍스트 격리다.** 총괄 세션이 모든 구현 세부를 떠안으면 컨텍스트가 오염되어 계획 능력이 떨어진다. 작업 세션은 자기 명세만 알면 되고, 끝나면 버려진다.

**총괄 세션이 결정권자가 아니다.** 총괄은 소유자의 결정을 명세로 번역하는 역할이다. 명세에 무언가가 있다는 것은 그것이 승인됐다는 뜻이 아니다 — 이 구분이 §5의 근거다.

---

## 2. 사이클

```
소유자 + 총괄  →  계획 수립
총괄           →  수용 기준 초안
소유자         →  수용 기준 승인          ← 유일한 승인 지점 (§3)
총괄           →  명세 + 프롬프트 작성
소유자         →  작업 세션 생성, 프롬프트 전달
작업 세션      →  구현 → QUESTIONS.md
총괄           →  게이트 (§5)
총괄 + 소유자  →  증류 (§6) → 브랜치 머지 → 브랜치 삭제
                                            → 명세·QUESTIONS 자동 소멸
```

병렬 가능한 웨이브는 워크트리로 나눈다(§7). 웨이브 내 모든 작업 세션이 끝나면 소유자가 총괄에게 알리고, 게이트를 통과해야 다음 웨이브 계획으로 넘어간다.

**선택: 위임 직전 적대적 검토.** 명세를 별도 세션에 넘겨 "이 명세의 빈틈·모순·과잉을 찾아라"를 시킬 수 있다. 총괄이 자기 명세를 스스로 검토하는 것보다 낫다. **아무 웨이브에서나 도입해도 된다** — 워크플로우 요소는 중간에 넣어도 아무것도 깨지지 않는다.

---

## 3. 승인 표면은 명세가 아니라 **수용 기준**이다

**소유자는 명세 전문을 읽지 않는다.** 8개 모듈 × 수십 페이지를 매 웨이브마다 정독하는 것은 가능한 일이 아니고, 그것을 요구하는 워크플로우가 틀렸다.

대신 **수용 기준을 승인한다.** 명세는 이미 자기 자신을 요약해 놓았다 — `E-1~7`, `AW-1~10`, `S-1~8`은 10줄 안팎이다.

**규칙:**

- 총괄은 명세를 쓰기 **전에** 수용 기준 목록을 한국어 한 줄씩 뽑아 소유자에게 제출한다.
- 소유자가 승인한 수용 기준이 **계약**이다.
- 명세 전문은 작업 세션용이다. 소유자는 안 읽는다.
- 승인되지 않은 수용 기준은 명세에 들어갈 수 없다.

> **왜:** MVP에서 1vs1 재생이 "단일 보드 + 플레이어 셀렉터"로 구현됐다. 소유자는 요청한 적이 없다. 총괄이 명세에 썼고 → 수용 기준 ID로 승격됐고 → 테스트가 통과했고 → 게이트 11항을 전부 통과했다. AW 수용 기준 10줄 중에 그 한 줄이 있었다면 소유자는 0.5초 만에 잡았다.

## 4. 파일의 수명

| 파일 | 사는 곳 | 죽는 때 |
|---|---|---|
| **명세** (`spec.md`) | **작업 브랜치 안** | 머지와 함께 자동 소멸 |
| **QUESTIONS.md** | 작업 브랜치 안 | 게이트에서 처분 후 소멸 |
| **계약** (스키마·타입·상수) | `packages/types` | 코드가 살아있는 한 |
| **결정** | `docs/DECISIONS.md` | 영구 |
| **규범 참조** (Tetr.io 룰 원문) | `docs/specs/` | 영구 |
| **리서치 로그** | `docs/research/YYYY-MM-DD-*.md` | 영구 (날짜 박아서) |

**명세를 `main`에 커밋하지 않는다.** 브랜치에만 두면 머지 후 알아서 사라진다.

> **왜:** MVP 명세 8개가 구현 완료 후에도 `docs/specs/`에 남았다. 코드와 일치하도록 사후 정합까지 했다. **코드와 완벽히 일치하는 문서는 정보량이 0이고, 이후로는 동기화 비용만 발생하는 부채다.** 그리고 사람은 낡은 문서를 무시하지만 **에이전트는 읽고 믿는다.** 구현이 끝난 명세가 저장소에 있으면 다음 세션이 그것을 현재 명세로 읽는다.

**따름정리 — 개정 이력 절을 두지 않는다.** 명세는 곧 삭제되므로 개정 이력을 남길 대상이 아니다. 사후 정합 작업 자체가 사라진다(§8).

**리서치 로그에는 반드시 날짜를 파일명에 박는다.** "2026-07-10 시점의 triangle 실측"은 역사적 사실이라 거짓이 될 수 없다. 현재형으로 쓰인 명세("엔진 API는 X다")는 언제든 거짓이 된다.

## 5. 게이트 체크리스트

총괄이 검사하고 소유자가 판정한다. **12항 전부 통과해야 웨이브가 끝난다.**

### 정적 검사

1. 수용 기준 ID가 테스트 이름에 전부 존재하고 통과 (`describe("E-1 …")` 대조)
2. `pnpm lint && typecheck && test` 루트 통과
3. 담당 패키지 밖 수정 없음 (특히 `types`·`docs`·루트 설정)
4. 승인 외 런타임 의존성 없음 / 의존 방향 위반 없음
5. 결정론 패키지에 `Math.random`·`Date` 없음 / 시크릿 노출 없음
6. `QUESTIONS.md` 잔여 없음 (또는 명시적 보류 사유)
7. 스키마 필드 철자가 계약과 일치
8. README 사용 예시 갱신

### 실동작 검사

9. `node tools/check-acceptance.mjs` 통과
10. **push 후 CI green 확인** — 로컬 green ≠ CI green
11. **웹 웨이브는 총괄이 `astro preview` 실조작** — mock 통과 ≠ 실동작

### 의도 검사

12. **이 웨이브 산출물 중 소유자가 요청한 적 없는 기능이 있나?**
    - "명세에 있었다"는 근거가 아니다. **명세도 총괄이 썼다.**
    - 소유자만 답할 수 있다. 총괄이 대신 판정할 수 없다.

> **왜 9~11:** W3에서 업로드 버튼 부재·tick 미배선이 게이트를 통과했다. 1~8은 전부 정적 검사(ID 존재·경계·grep·철자)라 **mock 위에서 통과 가능**했다. 실측 앵커에 묶이지 않은 조립 계층을 못 잡았다.
>
> **왜 12:** 1~11은 전부 "명세를 지켰나"를 묻는다. 그런데 명세는 총괄이 썼다. **순환이다.** "명세가 옳았나"를 묻는 항목이 하나도 없었고, 그것이 1vs1 단일 보드·op-log 페이지 모델·조작 재생 애니메이션의 공통 원인이다. 이것은 9~11로는 잡히지 않는다 — 그 셋은 전부 정상 동작했다.

## 6. 증류 — 게이트의 마지막 단계

명세와 QUESTIONS는 곧 사라진다. **사라지기 전에 오래 살 것을 꺼낸다.**

- **계약이 생겼거나 바뀌었나?** → `packages/types` 갱신
- **되돌리기 어려운 선택을 했나?** → `docs/DECISIONS.md` 추가
- **QUESTIONS.md의 각 항목** → 결정 / 이슈 / 코드 TODO 중 하나로 처분. **처분 안 된 항목이 남으면 게이트 통과 불가.**

> **왜:** MVP에서 QUESTIONS.md 12개가 처분되지 않은 채 쌓였다. 게이트 6항("잔여 없음")이 있었는데도 그랬다 — 잔여 여부만 봤지 **어디로 갔는지**를 안 봤기 때문이다.

## 7. 워크트리

**bare 저장소 구조를 쓴다.** 작업 디렉터리 밖에 워크트리가 놓이므로 gitignore 고민이 없고, 총괄과 작업 세션의 디렉터리가 깨끗하게 분리된다.

병렬 세션이 늘어나면 `gtr` 같은 워크트리 러너를 재검토한다. 현재 규모에서는 도입하지 않았다.

---

## 8. 구현이 명세에서 벗어났을 때

**명세를 고치지 마라.** 대신 묻는다:

| 이탈의 성격 | 처분 |
|---|---|
| **계약 변경** (스키마·API·상수) | `packages/types` 수정. 실행되므로 썩지 않는다. |
| **구현 세부** | **아무것도 하지 않는다.** 명세는 곧 삭제될 파일이다. |
| **수용 기준 변경** | 소유자 재승인 필요. 총괄 단독 불가. |

> **왜:** MVP 명세 8개 중 5개가 개정 이력을 달고 사후 정합됐다. 문서 스스로 "구현·테스트가 고정한 잠정안의 승인"이라고 적었다. 구현이 명세를 이기는 것은 정상이다 — 짜다 보면 알게 되는 것이 있다. **문제는 그 정합 작업 전체가 낭비였다는 것이다.** 곧 지울 파일을 코드에 맞추느라 웨이브마다 시간을 썼다.

## 9. 소유자 규칙

이 절은 총괄이 아니라 **소유자 자신에게 적용되는 규칙**이다.

### 9-1. 이해 못 하면 승인하지 않는다. 예외 없음

질문 하나의 비용은 토큰 몇백 개다. 이해 못 한 승인의 비용은 op-log 페이지 모델, 조작 재생 애니메이션, 이해 못 한 채 내린 fixture 공개 결정이다. **100배 차이다.**

**물어보는 것이 총괄 세션의 컨텍스트에 부담이면 — 새 세션을 열어서 물어라.** 그것이 원래 습관이다. 총괄에게 물을 필요가 없다.

> **왜:** *"왜 그걸 정해야 하는지, 비공개로 그냥 하면 되는 거 아닌지 **컨텍스트 오염될까 봐 안 물어봄**"* — 컨텍스트 위생이 이해 못 한 승인의 알리바이로 변질된 실제 사례다. (결과적으로 그 결정은 옳았으나, 옳았던 것은 운이다.)

### 9-2. 에이전트가 정한 마감은 존재하지 않는다

마감은 소유자가 정한다. 에이전트가 만든 긴박감("이 웨이브 전에 정해야 합니다")은 근거가 아니다.

> **왜:** fixture 공개 3안이 매 답변마다 재촉과 함께 제시됐고, 소유자는 차이를 이해하지 못한 채 "에이전트가 정한 마감 웨이브" 전에 공개를 결정했다. 이 프로젝트에서 유일하게 **되돌리기 어려울 뻔했던** 사건이다.

### 9-3. 에이전트가 3안을 들고 오면, 4번째 선택지는 "지금 안 정한다"이다

선택지 목록은 총괄이 만든 것이다. 목록 자체를 거부할 수 있다.

## 10. `v1` / `v2` 라벨 금지

MVP 문서 전반에 `v1`/`v2`가 쓰였으나 **"v1이 무엇을 만족하면 끝인지" 정의된 적이 없다.** 라벨만 있고 게이트가 없었다. 결과적으로 `v2`는 "안 함"의 완곡어법으로 기능했고, 소유자는 그것을 "언젠가 함"으로 읽었다.

**대신 두 개의 독립된 축으로 나눈다:**

| 축 | 질문 | 성격 |
|---|---|---|
| **동결** | 스키마·링크 형식에 흔적을 남기나? | **지금 결정. 되돌릴 수 없음.** |
| **순서** | 언제 구현하나? | Milestone. **언제든 바꿔도 됨.** |

동결 대상은 웨이브 스펙으로, 순서는 GitHub Milestone으로 간다. **`v2`라고 적힌 것 중 상당수가 사실 동결 창 안에 있었다** — 라벨이 위험을 가리고 있었다.

**되돌릴 수 없는 것의 판별:** 마이그레이션이 가능한가? 이 프로젝트는 모든 Gist가 소유자 부계정 소유이고 PAT가 있으므로 **대부분 마이그레이션이 가능하다.** 진짜 되돌릴 수 없는 것은 **남의 손에 나간 것** — 공유된 링크의 형식, 그 링크에 들어간 id, 남의 브라우저 쿠키에 있는 시크릿뿐이다.

**창이 닫히는 시점은 기술이 아니라 행동이다: 첫 링크를 남에게 공유하는 순간.**

## 부록 A. 명세 골격

MVP 8개 명세가 공유한 구조다. 유지한다.

```
# Tetorial 명세: <역할> (@tetorial/<pkg>)

> 상태: · 관련 모듈: …
> 역할: **한 문장**
> (해당 시) 실증/전제 근거

## 1. 경계 / 책임
   무엇을 하고 무엇을 하지 않는가. 예외 없이 첫 절.

## 2~N. API · 규범 규칙 · 데이터 모델
   모듈별 가변.

## 수용 기준 (작업 세션 완료 조건)
   - **<ID> <이름>**: <검증 내용>
   ※ 소유자가 승인한 것만. 여기 없는 기능은 만들지 않는다.

## 참고자료
   실측 로그·fixture·DECISIONS의 D-번호 역인용
```

**불변 요소:**

- **모듈별 고유 ID 접두** — `E`(engine) `A`(adapter) `W`(worker) `RD`(renderer) `RT`(replay-tetrio) `I`(input) `S`(sim) `AW`(apps-web). 수용 기준마다 배정하고 테스트 이름에 그대로 쓴다.
- **`규범(規範)` 태그** — 임의 해석 금지 지점 표시. 오프바이원 방지 수식, 변환 표, 분기 의미론 등.
- **실측이 추측을 이긴다** — 근거는 리서치 로그와 실물 데이터를 인용한다. 심판 규칙을 명시한다: *"다르면 triangle이 이긴다, E-4가 심판"*.
- **`DECISIONS.md`의 D-번호를 역인용** — 명세가 결정 로그에 못박히도록.

## 부록 B. 프롬프트 골격

### 공통 헤더 (전 세션 동일)

```
[착수 전 필독]
1. docs/DECISIONS.md
2. docs/CONVENTIONS.md
3. 담당 명세

[절대 규칙]
```

```
- 명세: <경로> (+ 부록·데이터)
- 참고: <리서치 로그·fixture·devDep 허용 범위>
- 주의: ①②③④        ← 함정 목록
- 완료 기준: <ID 범위>
```

**`주의` 필드가 프롬프트의 핵심이다.** 프로퍼티명 함정, 부호 규약(카운터 `-1`), 결정론 금지(랜덤·시각), 심판 규칙을 압축해 넣는 자리다.
